# Issue Planning Skill

## 目的

この Skill は、LabFlow iOS の機能や改善案を GitHub Issue に分解するときに使用する。

大きな機能をそのまま実装するのではなく、1人アジャイル開発で扱いやすい単位に分割する。

---

## 使用する場面

この Skill は、以下の作業時に使用する。

- 新機能を Issue に分解するとき
- 大きすぎる Issue を分割するとき
- Sprint Planning を行うとき
- MVP に必要なタスクを整理するとき
- Claude に実装依頼しやすい単位へ分解するとき
- 完了条件を定義するとき

---

## 基本方針

- 1 Issue は 1〜3時間程度で完了できる粒度にする
- 1 Issue には目的・背景・実装内容・完了条件を書く
- 1 Issue に複数の大きな機能を混ぜない
- UI、モデル、保存処理、編集処理などを分ける
- Claude に依頼しやすいように、作業範囲を明確にする
- 完了条件を満たせば Done にできる状態にする

---

## Issue の良い粒度

悪い例：

```text
タスク管理機能を作る
```

理由：

- 範囲が広すぎる
- 何をもって完了か分からない
- PR が大きくなりすぎる
- Claude に依頼したときに変更範囲が広がりやすい

良い例：

```text
TaskItem モデルを作成する
TaskListView を作成する
TaskCreateView を作成する
タスクの状態変更を実装する
SwiftData でタスクを保存する
```

理由：

- 作業範囲が明確
- 完了条件を書きやすい
- PR が小さくなる
- Claude に依頼しやすい

## Issue テンプレート

Issue は以下の形式で作成する。

```md
## 目的

このIssueで実現したいことを書く。

## 背景

なぜこの機能が必要なのかを書く。

## 実装内容

- [ ] 
- [ ] 
- [ ] 

## 完了条件

- [ ] Xcodeでビルドできる
- [ ] 画面上で動作確認できる
- [ ] 既存機能を壊していない
- [ ] 必要に応じてPreviewで確認できる

## 補足

関連する画面、モデル、参考資料などを書く。
```

---

## Issue 作成時の観点

Issue を作るときは、以下を確認する。

- この Issue の目的は一言で説明できるか
- 作業範囲が広すぎないか
- 1〜3時間で終わる見込みがあるか
- UI とデータ保存が混ざりすぎていないか
- 完了条件が具体的か
- Claude に実装依頼したときに迷わない内容か
- 既存機能への影響範囲が分かるか

---

## 機能を Issue に分解する手順

大きな機能は、以下の順番で分解する。

1. モデルを定義する
2. 一覧画面を作る
3. 作成画面を作る
4. 詳細画面を作る
5. 編集機能を作る
6. 削除機能を作る
7. 保存処理を追加する
8. エラー表示を追加する
9. 空状態を追加する
10. UI を改善する

## Issue の完了条件例

### モデル作成 Issue

```md
## 完了条件

- [ ] モデルが定義されている
- [ ] 必要なプロパティが含まれている
- [ ] SwiftData で扱える形になっている
- [ ] Xcode でビルドできる
```

### 一覧画面 Issue

```md
## 完了条件

- [ ] 一覧画面が表示される
- [ ] 仮データまたは保存済みデータが表示される
- [ ] 空状態が表示される
- [ ] Xcode でビルドできる
- [ ] Preview または Simulator で確認できる
```

### 作成画面 Issue

```md
## 完了条件

- [ ] 入力フォームが表示される
- [ ] 必須項目が未入力の場合は保存できない
- [ ] 保存ボタンが機能する
- [ ] 保存後に一覧へ戻る
- [ ] Xcode でビルドできる
```

### 保存処理 Issue

```md
## 完了条件

- [ ] 新規作成したデータが保存される
- [ ] アプリ再起動後もデータが残る
- [ ] 既存データを壊していない
- [ ] エラー時の挙動が最低限考慮されている
```

```text
条件：
- 1 Issue は 1〜3時間程度の粒度にする
- SwiftUI + MVVM + SwiftData を前提にする
- 各 Issue に目的・実装内容・完了条件を含める
- 無理に大きな Issue にまとめない
```

## Sprint Planning の進め方

1週間の Sprint では、無理に多くの Issue を詰め込みすぎない。

目安：

```text
小さい Issue：3〜5個
中くらいの Issue：2〜3個
大きい Issue：分割してから着手
```

Sprint の目標は一言で定義する。

例：

```text
Sprint 2 の目標：
タスクを作成・一覧表示できる状態にする。
```

---

## Sprint に入れる Issue の選び方

優先度は以下の順に考える。

1. MVP に必要なもの
2. 他の機能の土台になるもの
3. 画面として早く確認できるもの
4. 保存・編集・削除など基本機能
5. UI 改善
6. 将来的な拡張機能

---

## Issue 分解時のチェックリスト

- [ ] 1つの Issue が大きすぎない
- [ ] 目的が明確である
- [ ] 背景が書かれている
- [ ] 実装内容がチェックリスト化されている
- [ ] 完了条件が具体的である
- [ ] UI と保存処理が必要以上に混ざっていない
- [ ] Claude に依頼しやすい粒度である
- [ ] PR が大きくなりすぎない見込みである
